# QuickCapture - Product Requirements Document
Version 2.0 | Date: November 2024
Multi-Platform Edition: Phone, Wear OS, TV, Automotive, XR

## Executive Summary
QuickCapture is a privacy-first, multi-platform quick note capture ecosystem that enables users to instantly save thoughts, voice recordings, and meeting notes from ANY device they interact with - phone, watch, TV, car, or XR headset. All processing happens on-device by default, with seamless sync between platforms and optional cloud features.

## 1. Product Vision

### Vision Statement
To create an omnipresent capture layer across all computing surfaces, ensuring no thought is ever lost regardless of which device is nearest, while maintaining absolute privacy.

### Mission
Enable frictionless thought capture across every digital touchpoint in a user's life while maintaining complete data sovereignty and privacy.

### Core Principles
- **Ubiquity**: Available on every screen and device
- **Speed**: Capture in under 2 seconds on any platform
- **Privacy**: Data never leaves the user's device ecosystem without explicit permission

# QuickCapture - Technical Architecture Document
Version 2.0 | Multi-Platform Architecture

## 1. Architecture Overview

### System Design Principles
- **Kotlin Multiplatform Core**: Shared business logic across all platforms
- **Platform-Native UI**: Each platform uses its native UI framework
- **Local-First Storage**: Each device maintains local database
- **Mesh Sync Network**: Devices sync peer-to-peer when possible
- **Privacy by Design**: End-to-end encryption available, no required cloud

### Architecture Layers

#### Layer 1: Shared Core (Kotlin Multiplatform)
- Domain models and entities
- Business logic and rules
- Sync protocol implementation
- Encryption and security
- Provider interfaces
- Data transformation logic

#### Layer 2: Platform Adaptation
- Platform-specific implementations
- UI/UX adaptations
- Hardware capability detection
- Platform service integration
- Input method handling

#### Layer 3: Device Services
- Background sync services
- Voice processing
- Notification handling
- Widget/Complication services
- Platform-specific storage

## 2. Multi-Platform Architecture

### Platform Deployment Strategy

#### Supported Configurations
1. **Standalone Mode**: Single device operation
2. **Companion Mode**: Device requires phone (Wear OS)
3. **Ecosystem Mode**: Full multi-device sync
4. **Family Mode**: Multiple users on shared devices (TV)
5. **Offline Mode**: No sync, local only

### Platform Capability Matrix

| Capability | Phone | Wear OS | TV | Auto | XR |
|-----------|-------|---------|-----|------|-----|
| Local Storage | Full | Limited | Full | Limited | Limited |
| ML Processing | Yes | Via Phone | Yes | No | Via Phone |
| Background Sync | Yes | Limited | Yes | No | Limited |
| Multi-User | No | No | Yes | Yes | No |
| Voice Input | Yes | Yes | Yes | Yes | Yes |
| Visual Input | Yes | No | No | No | Yes |
| Biometric Context | Limited | Yes | No | No | Limited |

### Data Flow Architecture

#### Capture Flow
1. **Input Layer**: Platform-specific input (voice, touch, gesture)
2. **Processing Layer**: Platform-optimized processing
3. **Storage Layer**: Local database write
4. **Sync Layer**: Queue for cross-device sync
5. **Notification Layer**: Update other devices

#### Sync Architecture
- **Primary Sync**: Phone acts as hub
- **Secondary Sync**: Direct device-to-device when available
- **Fallback Sync**: Optional encrypted cloud relay
- **Conflict Resolution**: Last-write-wins with version control

## 3. Platform-Specific Architectures

### Android Phone Architecture
- **Role**: Primary hub and configuration center
- **Storage**: Full SQLite database with all features
- **Services**: Background sync, notification management
- **UI Framework**: Jetpack Compose
- **Special Features**: Widget support, share intent receiver

### Wear OS Architecture
- **Role**: Companion device for quick capture
- **Storage**: Limited cache, sync with phone
- **Services**: Health service integration, complication provider
- **UI Framework**: Wear Compose
- **Special Features**: Always-on display, rotary input support

### Android TV Architecture
- **Role**: Family hub for shared and personal notes
- **Storage**: Full database with multi-user support
- **Services**: Ambient mode service, voice control
- **UI Framework**: Leanback/Compose for TV
- **Special Features**: Multi-user profiles, screensaver mode

### Android Automotive Architecture
- **Role**: Safe capture while driving
- **Storage**: Limited cache for safety
- **Services**: Google Assistant integration, location service
- **UI Framework**: Car App Library
- **Special Features**: Driving state detection, voice-only mode

### XR/AR Architecture
- **Role**: Spatial and visual note capture
- **Storage**: Limited local, sync required
- **Services**: Spatial tracking, gesture recognition
- **UI Framework**: Platform-specific AR SDK
- **Special Features**: 3D note placement, world anchoring

## 4. Data Architecture

### Storage Strategy

#### Local Storage Hierarchy
1. **Primary Storage**: Full database on phone/TV
2. **Cache Storage**: Limited storage on Watch/Auto/XR
3. **Temporary Storage**: Session-only for sensitive data
4. **Archive Storage**: Optional cold storage on phone

#### Database Schema Approach
- **Universal Schema**: Core tables on all platforms
- **Platform Extensions**: Device-specific metadata tables
- **Sync Metadata**: Version control and conflict tracking
- **Encryption Layer**: Optional E2E encryption

### Sync Protocol Design

#### Sync Methodology
1. **Device Discovery**: Bluetooth, WiFi Direct, or Cloud
2. **Authentication**: Device pairing and trust establishment
3. **Differential Sync**: Only changed data transmitted
4. **Conflict Resolution**: Platform priority and timestamps
5. **Verification**: Checksum and acknowledgment

#### Platform Sync Priority
1. Phone (highest authority)
2. Tablet
3. TV
4. XR
5. Watch
6. Auto (lowest authority)

## 5. Security Architecture

### Privacy Framework

#### Data Classification
- **Level 0**: Public (templates, settings)
- **Level 1**: Personal (unencrypted notes)
- **Level 2**: Private (encrypted notes)
- **Level 3**: Sensitive (biometric, health data)

#### Platform Security Features
- **Phone**: Full encryption, biometric unlock, secure element
- **Watch**: PIN protection, wrist detection
- **TV**: User profiles with PIN
- **Auto**: Driver verification, movement restrictions
- **XR**: Gaze authentication, spatial privacy zones

### Encryption Strategy
- **At Rest**: AES-256 local encryption
- **In Transit**: TLS 1.3 for network communication
- **End-to-End**: Optional Signal protocol for sync
- **Key Management**: Platform-specific secure storage

## 6. Integration Architecture

### Provider System

#### Provider Interface Design
- Platform-agnostic provider interface
- Capability detection per platform
- Graceful degradation for limited devices
- Queue management for offline operation

#### Supported Providers by Platform
- **Phone**: All providers (Obsidian, Notion, etc.)
- **Watch**: Limited to simple text providers
- **TV**: Read-heavy providers only
- **Auto**: Voice-note providers only
- **XR**: Visual-capable providers

### External Service Integration

#### Voice Processing
- **Primary**: On-device Whisper model (Phone, TV)
- **Secondary**: Platform speech services
- **Fallback**: Optional cloud transcription
- **Platform Routing**: Watch/Auto ‚Üí Phone processing

#### AI Services (Optional)
- **Trigger**: User-initiated only
- **Processing**: Always on phone or TV
- **Privacy**: Clear consent and data handling
- **Platforms**: Available on capable devices only
